## 4. Euler 7（10001番目の素数）の最適化ドラマ

Project Euler の問題7（10001番目の素数を求める）が、AI の最適化能力を示す良い指標になっている。コンテキストオーバーフローの要約から追跡できる:

```
Session初期   1.34s（libjq フォールバック） → jq は 1.49s（すでに jq より速い）
Session ?     0.49s（ネイティブ実行化）       2.7x speedup
Session ?     0.421s
Session ?     0.285s                         eval_one fast path 導入
Session ?     0.104s                         4x more
Session ?     0.087s
最終          0.085s                         jq比 17.5x
```

1.34秒 → 0.085秒。同じプログラムを16倍速くした。しかも問題を解くアルゴリズム自体は変えていない — 実行エンジンの最適化だけでここまで来ている。

## 5. 失敗と撤退

成功ばかりではない。コンテキストオーバーフローの要約には、失敗して撤退した最適化の記録もある:

- **Closure-based function calls**: 試したが31%のリグレッション。即撤退
- **Null-input JIT compilation**: `-n` モードで JIT を有効化したら P1 が 3.8ms → 407ms に劣化。コンパイルオーバーヘッドが支配的になるケース
- **memchr SIMD**: 短いキー（1〜4バイト）では SIMD のセットアップコストが逆効果。スカラーのバイトループの方が速い
- **Rc\<String\> → Rc\<str\>**: Value 型の変更を試みて途中で断念（UNCOMMITTED として記録）
- **resolve_closure_refs at push time**: 4〜8%のオーバーヘッド増加。リバート
- **Numeric counter fast path**: select_prefix 最適化をバイパスしてしまい、リグレッション

AI は「やってみてダメなら戻す」を人間の介入なしに繰り返している。

## 7. 126回のコンテキストオーバーフローで何が起きていたか

各オーバーフローは1つの「セッション」に相当する。要約から見える進化のフェーズ:

| Overflow # | 主なテーマ |
|---|---|
| 1〜3 | eval_one fast path の導入、AllShort/AnyShort 最適化 |
| 4〜6 | Euler ベンチマーク統合、eval_one の全式への展開 |
| 7〜9 | mimalloc、mmap、Rc\<str\> の試行錯誤 |
| 10〜13 | CompactString、Vec-based ObjMap、push_compact_line |
| 14〜19 | Number parsing fast path、Collect+AllShort fusion、Euler 7 の 421ms→85ms |
| 20〜26 | JIT peephole、fast-float、inline 最適化の限界探索 |
| 27〜31 | JIT operation fusion、Raw byte passthrough 発明 |
| 32〜37 | Raw byte fast paths の体系的展開（7パターン） |
| 38〜40 | O(n²) → O(n) 問題の修正、TakeVar/MoveToVar |
| 41〜以降 | 正規表現 JIT、フォーマット最適化、セマンティック IR 書き換え |

特に注目すべきは Overflow #27〜31 あたり。ここで「JSON をパースしないで処理する」という Raw byte fast paths のアイデアが生まれている。これがプロジェクト全体で最もインパクトの大きい最適化クラスになった。

## 9. 人間の介入パターン

人間が技術的フィードバックを送ったのは実質3回:

1. **O(n²) 問題の報告**（3月8日）: 別セッションで発見された性能問題を、人間がコピペで伝えた。これは「AI-A の発見を人間が AI-B に伝達する」パターン
2. **floor リグレッション指摘**（3月12日）: ベンチマーク結果を人間が目視で確認し、リグレッションを1行で指摘
3. **カラー出力レビュー**（3月17日）: 実装済みのコードに対する詳細なバグレポート

それ以外は「続けて」「再開して」「push」のみ。人間は AI のマネージャーとして機能していて、実装には一切関与していない。

## 10. 2つの AI の知見の橋渡し

3月8日のメッセージが特に興味深い:

> "もう改善済みかもしれないけど、**別セッションで**こんな指摘があった"

この「AI 間の知見の橋渡し」は、複数の AI エージェントを並行運用する際の典型的なパターンになりそう。
