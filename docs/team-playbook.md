# Claude Code を長時間自律駆動させるためのプレイブック

jq-jit の開発ログ（11日間・595コミット・6万行）から抽出した、チームで使える実践知見。

---

## 1. 自律駆動を成功させた5つの仕掛け

jq-jit では AI が126回コンテキストオーバーフローしながら自律的に動き続けた。これを支えた仕組みは以下の5つ。

### (1) CLAUDE.local.md に「判断を求めるな」と書く

```markdown
## Rules
- 判断を求めず、全て自律的に遂行する
- 適切にコミットする
- コミット前に `cargo build --release` で警告ゼロを確認する
```

**ポイント**: 「相談してね」だと AI は毎回聞いてくる。「判断を求めるな」と明示すると、自分でベストを判断して進む。ただしこれは**安全な境界**とセットにすること（次項）。

### (2) 安全な境界を明示する

```markdown
- PC ローカルの作業にとどめる。外部に影響を与えない（push、Issue 作成、外部 API 呼び出し等は禁止）
```

「自律的にやれ」と「ここだけは絶対やるな」の両方を書く。自由度と安全性のバランス。jq-jit では push は人間が手動で行い、AI は一切外部に影響を与えなかった。

### (3) 検証可能なゴールを設定する

```markdown
- 100% jq 互換（509/509 テスト通過）
- JIT コンパイルにより jq より速い
```

「良いコードを書け」ではなく「509/509 を通せ」「jq より速くしろ」。数値で測れるゴールがあると、AI は自分でフィードバックループを回せる。

実際の jq-jit では、AI は以下のサイクルを人間の介入なしに回し続けた:

```
コード変更 → cargo build（1,353回）→ cargo test（1,152回）→ ベンチマーク（909回）→ コミット（498回）
```

1コミットあたり平均 **ビルド+テスト 5回、ベンチマーク 1.8回**。壊れたら自分で直す、遅くなったら自分で戻す。

### (4) 進捗の可視化を仕込む

```markdown
jq-jit-history.html を作成し、やった作業を時系列で報告する。
ポート番号 18080 でローカルネットワークに配信する。
```

AI が何をやっているかを**人間が別端末から見られる**ようにした。これにより:
- 行き詰まっていたら介入できる
- 成果が出ていれば放置できる
- 寝てる間の作業も翌朝確認できる

### (5) コミット粒度を AI に任せる

「適切にコミットする」とだけ書いて、粒度は指定しなかった。結果:
- コミット間隔の中央値は **9.4分**
- 1つの最適化 = 1コミットの粒度で、後から git log で追跡可能
- ベンチマーク結果がコミットメッセージに含まれる（例: `33.6s→0.34s (99x)`）

## 2. 人間の介入パターン: 3種類だけ

11日間で人間が送った実質的なメッセージは25件。分類すると3種類に集約される。

### 種類A: 方向付け（最初の1回）

> "任せます。自律的に動き続けて、可能な限りパフォーマンスを最適化してください"

ゴールを設定したらあとは任せる。

### 種類B: 継続指示（6回）

> "続けて" / "再開して" / "PUSHなどしただけ、続けて"

コンテキストオーバーフローやセッション中断の後に、1〜2語で再開させる。

### 種類C: ピンポイント技術フィードバック（3回）

1. **別 AI の知見を伝達**: 別セッションの Claude が発見した O(n²) 問題をコピペで伝えた
2. **リグレッション指摘**: 「floor が遅くなっている」の1行
3. **コードレビュー**: カラー出力実装のバグ3件を具体的に報告

**学び**: 人間が介入するのは AI が自力では気づけないことだけ。別の AI からのクロス情報、目視でしか気づかないリグレッション、リリース品質のコードレビュー。

## 3. 効果的な指示の書き方

jq-jit で機能した指示と、そうでなかった指示の比較。

### 効果的だった指示

| 特徴 | 例 |
|---|---|
| 検証可能なゴール | "509/509 テスト通過" |
| 制約の明示 | "外部に影響を与えない" |
| 自律性の明示的な付与 | "判断を求めず、全て自律的に遂行する" |
| 具体的な検証手段 | "cargo build --release で警告ゼロを確認" |
| 観測手段の設計 | "history.html をポート18080で配信" |

### 最初の指示に含めるべき要素

jq-jit の初期指示（最も構造化されていたもの）をテンプレートにすると:

```markdown
## ゴール
（検証可能な完了条件）

## 注意事項
- 判断を求めず、全て自律的に遂行してください
- （安全境界: やってはいけないこと）
- 適切にコミットしてください
- （セッション管理への言及）

## 進捗共有
（人間が非同期で確認できる仕組み）
```

## 4. 複数 AI エージェントの並行運用

jq-jit では同じマシンで2つの Claude Code セッションが並行して走っていた。

```
ターミナル1: セッション A（アプローチ1）
ターミナル2: セッション B（アプローチ2）
```

### 得られた知見

**アプローチの競争が有効**。同じゴール（509/509通過 + jq より速い）に対して異なる設計で並行開発し、結果が良い方を採用した。初期段階で 4〜6倍だったものが、最終的に 8〜182倍まで到達した。

**AI 間の知見伝達は人間が行う**。あるセッションの Claude が見つけた O(n²) パターンを、人間がコピペで別のセッションの Claude に伝えた。AI 同士は直接通信できないので、人間がメッセンジャーとして機能する。

**失敗したアプローチも価値がある**。jq-jit0 の CPS 方式の設計上の壁（ジェネレータのスカラーコンテキスト問題）は、jq-jit の設計で回避すべき教訓になった。

### チームでの応用

- 設計が分かれる場面で、2つの Claude Code セッションに別アプローチを試させる
- 片方の発見を人間が他方に伝える「ブリッジ」役を意識する
- 両方の結果を比較して判断するのは人間の仕事

## 5. 長時間セッションの実務的な注意点

### コンテキストオーバーフローへの対処

jq-jit のメインセッションは126回オーバーフローした。Claude Code は自動要約して継続するが、**メモリシステム（memory/）** に重要な状態を保存しておくと精度が上がる。

jq-jit では以下がメモリに保存されていた:
- アーキテクチャの設計判断
- Value enum のメモリレイアウト（バイトオフセットまで）
- 適用済みの最適化リスト
- ベンチマーク結果の推移
- プロファイリングの知見

### history.html パターン

「進捗を HTML で書いて HTTP 配信しろ」は汎用的に使える。AI の作業を別端末・スマホから確認できる。

```markdown
## 進捗共有
{project}-history.html を作成し、やった作業を時系列で報告する。
ポート番号 18080 でローカルネットワークに配信する。
```

### push のタイミング

jq-jit では push は人間が手動で行った（AI は禁止）。これにより:
- AI が意図しないコードを外部に公開するリスクをゼロにした
- 人間が push 前にざっと diff を見る機会を確保した
- "push" → "続けて" が自然な介入ポイントになった

## 6. AI が自律的に発見した最適化手法

人間が教えたわけではないのに、AI が自分で発見・実装した技法のうち、汎用的に応用できるもの:

### プロファイル駆動の最適化サイクル

AI は macOS の `sample` コマンドで419回プロファイリングを実行した。以下のサイクルを自律的に回し続けた:

```
プロファイル → ボトルネック特定 → 仮説 → 実装 → テスト → ベンチマーク → コミット
```

失敗した最適化（31%リグレッション等）は即座にリバートし、別のアプローチを試している。

### 「試してダメなら戻す」の文化

ログに残っている失敗:
- Closure-based function calls → 31%リグレッション → リバート
- memchr SIMD → 短いキーで逆効果 → スカラーループに戻す
- Rc\<String\> → Rc\<str\> → 途中で断念
- Null-input JIT → 3.8ms→407ms に劣化 → リバート

AI は失敗を恐れない。テストスイートとベンチマークがあれば、大胆に試して客観的に判断できる。

## 7. 失敗はめちゃくちゃ成功: 「失敗ドキュメント駆動開発」

jq-jit の最大の知見は、**最初の失敗が成功の鍵だった**ということ。

### 実際に起きたこと

```
3月1日   オリジナル jq-jit を Claude が開発（CPS + コールバック方式）
          → 142 SKIP + 8 FAIL で行き詰まり
          → Claude が自発的に「何がダメだったか」をドキュメント化
```

AI が書いた失敗分析（`memory/architecture.md`）:

```markdown
## Old Design (CPS + Callback)

### Fatal Flaws
1. Generator-in-scalar-context: CPS+callback can't handle multi-value
   expressions in scalar positions (limit, first, last, nested generators
   in builtins). Root cause of 142 SKIP + 8 FAIL.
2. Object key ordering: BTreeMap (sorted) instead of insertion-order.
3. Path tracking: Incomplete - doesn't work with select/map.
4. Label/break: Unimplemented.
5. Try/catch: Incomplete generator interaction.
```

根本原因を1行で特定している: 「CPS + コールバックではジェネレータのスカラーコンテキストを扱えない」。

### 人間がやったこと

5日後、この1行だけを送った:

> "../jq-jit を把握して、設計上の課題を明らかにした上で、ゼロベースで構築してください"

「把握して」「課題を明らかにした上で」— つまり失敗ドキュメントを読め、という指示。

### AI がやったこと

失敗分析を読んだ上で:
- Fatal Flaw #1（ジェネレータ問題）を回避するため、CPS ではなくバイトコード VM 方式を採用
- Fatal Flaw #2（キー順序）を回避するため、BTreeMap ではなく IndexMap を採用
- **4時間で 509/509 達成**（前回は到達できなかった）

### なぜこれが重要か

普通、「失敗したから一から作り直そう」はコストが高い。でも AI を使うと:

1. **失敗のコストが極めて低い** — AI が書いたコードを捨てるのに心理的抵抗がない
2. **失敗の分析が自動で残る** — AI は「なぜダメだったか」を構造化して書き出す
3. **2回目は圧倒的に速い** — 失敗の知識があるから、同じ罠にはまらない
4. **失敗がなければ到達できない設計に辿り着く** — CPS を試さなければ「CPS ではダメ」とはわからない

### チームでの応用

**「AI に失敗させる」ことを恐れない。むしろ積極的に活用する。**

```
1回目: "○○を実装して"
  → 失敗 or 行き詰まり
  → AI が「何がダメだったか」をドキュメント化（勝手にやる）

2回目: "前回の失敗を踏まえて、ゼロから作り直して"
  → 失敗の知識を持った状態で再設計
  → 1回目では到達できなかった品質に到達
```

これは人間のエンジニアリングでは「プロトタイピング」と呼ばれるが、AI だとコストが桁違いに安い。6万行の JIT コンパイラを「まず1回作って捨てる」ことが現実的になる。

**1回目の失敗こそが、2回目の成功への最短ルート。**

## 8. コスト感覚のアップデート: 「捨てていい」という判断

jq-jit の事例で最も重要な気づきは、**ソフトウェア開発のコスト感覚が根本的に変わった**こと。

### 従来の常識

| 行動 | 従来のコスト感覚 |
|---|---|
| 6万行のコードを書く | 数ヶ月〜数年 |
| 動いているコードを捨てて作り直す | 非常に高い。避けるべき |
| 設計を検証するためにプロトタイプを作る | 数週間。リソースが許せば |
| 2つの設計を並行で試す | 贅沢。人が足りない |
| 失敗する | 時間・コスト・士気の損失 |

### AI 時代の現実

| 行動 | 実際にかかったコスト |
|---|---|
| 6万行のコードを書く | 11日間の Claude Code 利用料 |
| 動いているコードを捨てて作り直す | 「ゼロベースで構築して」の1行 |
| 設計を検証するためにプロトタイプを作る | 4時間（509/509 達成まで） |
| 2つの設計を並行で試す | ターミナルをもう1つ開くだけ |
| 失敗する | 次の成功への投資（失敗分析が自動で残る） |

### これが意味すること

**「もったいない」が判断を歪める最大の原因だった。**

人間が数ヶ月かけて書いたコードは捨てられない。サンクコストバイアスが働いて、根本的に設計が間違っていても修正で乗り切ろうとする。結果、技術的負債が積み上がる。

AI が書いたコードにはサンクコストがない。動いていても、もっと良い設計が見えたなら捨てればいい。捨てるコストは「もう1回作らせる時間」だけ。jq-jit の場合、それは4時間だった。

### チームで意識すべき判断基準の変化

| 従来の判断 | AI 時代の判断 |
|---|---|
| "動いてるから触るな" | "もっと良い設計が見えたなら作り直そう" |
| "プロトタイプを作る余裕はない" | "まず雑に作って学び、捨てて本番を作ろう" |
| "この設計で行くしかない" | "2つ試して良い方を採用しよう" |
| "失敗したら時間の無駄" | "失敗したら次のインプットが手に入った" |
| "慎重に設計してから書き始めよう" | "書いてみないとわからないことは書いて確認しよう" |

特に最後が重要。jq-jit では「CPS + コールバックでジェネレータのスカラーコンテキストが扱えない」という知見は、**実際に作ってみないと得られなかった**。机上の設計レビューでは見つからなかった可能性が高い。

**「まず作って、失敗して、学んで、捨てて、作り直す」が、AI 時代の最速ルート。**

## 9. チームへの展開チェックリスト

Claude Code を自律的に使う際に、最低限用意すべきもの:

- [ ] **CLAUDE.local.md** に自律性と安全境界を明記
- [ ] **検証可能なゴール**（テストスイート、ベンチマーク、lint 等）
- [ ] **進捗可視化の仕組み**（history.html、ログファイル等）
- [ ] **push / デプロイの手動ゲート**
- [ ] **コミットルール**（粒度は AI に任せ、ビルド+テスト通過を義務付け）

### やってはいけないこと

- [ ] 「相談して」と書く → AI が毎回聞いてきて自律性が失われる
- [ ] 安全境界なしで自律性を付与する → push や外部 API を勝手に叩くリスク
- [ ] テストスイートなしで最適化を依頼する → リグレッションを検出できない
- [ ] コンテキストオーバーフロー後に状況説明なしで「続けて」 → AI が文脈を失う
